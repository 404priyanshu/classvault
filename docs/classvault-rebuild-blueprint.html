01 · Rebuild Thesis

Rebuild Thesis

If I rebuilt ClassVault from scratch, I would not start with screens. I would start with the trust model: who can upload, who can see, who can verify, who can moderate, and what evidence the AI is allowed to use.

ClassVault is not just a notes app. The useful version is a campus knowledge network where study material becomes searchable, discussable, reusable, and AI-readable without becoming messy or unsafe. The current project already proves several important product primitives: notes, moderated uploads, full-text search, saves, ratings, comments, notifications, collections, study rooms, leaderboards, AI roadmaps, exam plans, and campus verification.

The smarter rebuild keeps the same core direction but changes the order. Build the durable spine first: institution, course, note version, file ledger, moderation event, AI usage event, document chunk, citation, and audit log. Then build the interface around those objects. That sequence makes AI features easier, search better, moderation safer, and paid institutional plans less painful later.

What I Would Keep

  • Next.js App Router, React, TypeScript, and pnpm. The current stack fits a product with server-rendered pages, route handlers, and client-side interactive islands.
  • Postgres with Prisma. The app needs relational integrity, full-text search, moderation history, usage counters, and later vector search. Postgres is the right center.
  • Zod at every boundary. AI output, upload metadata, form input, and route bodies all need runtime validation, not just TypeScript.
  • S3-compatible object storage. Keep signed upload and download URLs, a storage-key ledger, and file metadata verification.
  • Thin route handlers with server modules. The current repo already follows this pattern under lib/server/; preserve it.

What I Would Change

AreaCurrent shapeRebuild change
Data modelStrong notes core, institution and course exist but are still emerging.Make Institution, Course, Syllabus, NoteVersion, DocumentChunk, and AIUsageEvent first-class on day one.
AIDirect provider module with Gemini primary and OpenAI fallback.Use a provider abstraction plus AI SDK-style streaming, structured outputs, tool calls, retrieval, and per-institution quotas.
UploadsFile validation, storage keys, direct upload, moderation.Add async file processing: virus scan, text extraction, OCR fallback, duplicate detection, embeddings, and preview generation.
SearchPostgres full-text search plus filters and trending.Use hybrid search: full-text for exact course intent, vector search for semantic questions, and citation-backed AI answers.
Product flowMany features shipped in parallel.Sequence trust, corpus, search, and moderation before advanced community and AI features.
Core principle
Build ClassVault as a trusted study corpus first. The AI should be a layer over verified, searchable, permissioned material, not a magic text box sitting beside loose uploads.
02 · What To Do First

What To Do First

The first week should answer product and data questions, not animation and landing-page questions. A clean first week saves months of rewrites.

Day 1: Write The Product Contract

Create a one-page product contract before opening the editor. It should define the users, promises, non-goals, and trust rules. For ClassVault, the user roles are student, contributor, moderator, admin, and institution owner. The promise is simple: students find useful, campus-relevant material faster, and AI can explain or plan using the material with citations.

DecisionRecommended answerWhy it matters
Primary userStudent preparing for coursework or exams.Keeps the UI focused on search, saved notes, progress, and exam pressure.
Supply sourceStudent uploads, seeded official samples, and institution-curated material.Prevents the library from depending only on cold-start uploads.
Trust ruleFiles are not fully trusted until validated and reviewed.Protects the app from spam, copyrighted dumps, unsafe files, and low-quality notes.
AI ruleAnswers must cite notes, pages, chunks, or clearly say when no source exists.Prevents plausible but unsupported study advice.
Business ruleInstitutions pay for governance, analytics, quotas, and verified vaults.Gives the product a monetizable surface without charging every student first.

Day 2: Define The Non-Negotiable Flows

Map these flows before UI polish: sign up, verify campus, upload resource, moderate resource, search library, save note, ask note, report note, resolve report, and review institution analytics. If a screen does not support one of these flows, it is probably not part of the first build.

student signs up
  -> verifies campus email
  -> uploads file and metadata
  -> file enters processing and moderation
  -> approved note appears in course library
  -> classmates search, save, comment, rate, download
  -> AI can cite chunks from approved notes
  -> moderators handle reports and quality issues

Day 3: Create The Architecture Map

Days 4-5: Bootstrap The Repo

Use a boring setup. The product is already ambitious; the build system should not be. Start with pnpm, Next.js App Router, TypeScript strict mode, Tailwind, Prisma, Vitest, Playwright, ESLint, and a CI workflow that runs lint, typecheck, tests, Prisma validation, and build.

pnpm create next-app classvault --ts --app
cd classvault
pnpm add zod prisma @prisma/client pg @prisma/adapter-pg
pnpm add lucide-react motion @vercel/analytics @vercel/speed-insights
pnpm add -D vitest @playwright/test eslint-config-next tsx
pnpm prisma init

Do not add payment, chat, vector search, and background jobs on day one. Put placeholders in the architecture, not in the dependency list. Add the dependency when the first production path needs it.

When I Would Choose A Different Stack

If the product becomes mobile-first with heavy native features, move the backend into a separate API service and keep Next.js as the web client. If institution analytics becomes the main business, add a warehouse later rather than bending the app database into a BI system.

04 · Data Model

Data Model

A better ClassVault starts with a better schema. The database should describe the product you want in year two, not only the upload form you need in week one.

Start With These Core Entities

EntityPurposeBuild now?
UserStudent, moderator, admin, institution owner, verification state, role.Yes
InstitutionCollege identity, domain, plan, quotas, billing, analytics scope.Yes
CourseStable home for notes, PYQs, syllabus, followers, top contributors.Yes
SyllabusUnits, topics, outcomes, exam weight, prerequisite graph.Yes, simple version
NotePublished resource record with owner, course, metadata, status, ratings.Yes
NoteVersionFile replacement history, moderation per version, old storage keys.Yes
UploadedFileStorage ledger that prevents fake storage keys or wrong file metadata.Yes
DocumentChunkExtracted text, page, section, embedding id, moderation eligibility.Before AI Q&A
AIUsageEventUser, institution, feature, model, tokens or cost estimate, status.Before public AI
AuditLogSecurity and moderation-sensitive actions.Before staff tooling

Schema Pattern

Use Prisma for relational models and migrations. Use raw SQL where Prisma cannot express the exact Postgres feature cleanly, such as ranked full-text search, triggers for search vectors, and vector indexes. This is not a failure of Prisma; it is the right division of labor.

model Institution {
  id                    String   @id @default(cuid())
  name                  String   @unique
  slug                  String   @unique
  emailDomain           String?  @unique
  plan                  String   @default("free")
  quotaStorageBytes     BigInt?
  quotaAiRequestsMonth  Int?
  users                 User[]
  courses               Course[]
}

model Course {
  id            String   @id @default(cuid())
  institutionId String?
  code          String
  name          String?
  subject       String?
  notes         Note[]
  syllabusUnits SyllabusUnit[]

  @@unique([institutionId, code])
}

model Note {
  id              String      @id @default(cuid())
  courseId        String?
  ownerId         String
  title           String
  status          NoteStatus  @default(PENDING)
  currentVersionId String?
  versions        NoteVersion[]
  chunks          DocumentChunk[]
}

Indexes To Plan Early

  • Course lookup: institutionId, code and subject, semester.
  • Library browsing: status, createdAt, courseId, status, and tag join indexes.
  • Moderation queue: status, reviewedAt and reviewer history.
  • Trending: noteId, createdAt on download events.
  • Notifications: userId, readAt, createdAt.
  • Rate limits: key as primary id with window start and updated time.
  • AI retrieval: full-text vector, embedding vector index, noteId, pageNumber on chunks.

Migration Discipline

Use migrations as product history. Avoid changing a model and immediately coupling it to a large UI change. For risky schema work, ship in three moves: add nullable fields and backfill, read from the new fields with fallback, then enforce constraints after production data proves the migration.

Data model rule
Do not make AI features store their own invisible truth. Every AI answer should resolve back to Course, Note, NoteVersion, DocumentChunk, User, and Institution records.
05 · Trust, Uploads, And Moderation

Trust, Uploads, And Moderation

ClassVault lives or dies by trust. A fast upload flow is not enough; the app needs a defensible path from raw file to approved study resource.

Authentication And Campus Identity

Keep the custom cookie-session approach if campus verification, role bootstrapping, and institution scoping are central. Store only hashed session tokens, use HTTP-only cookies, and keep role checks server-side. Google OAuth and email OTP can coexist, but both should feed the same User and Session tables.

ConcernImplementation ruleTest
Session safetyHash raw session tokens before database storage.Unit test token hashing and expiry.
Campus accessRestrict or mark verified users by allowed email domains and verification codes.E2E sign-up plus verification path.
RolesUse server-side requireRole checks on every staff route.Route tests for student denial and moderator access.
Institution scopePass institution id into rate limits, queries, and analytics.Integration tests with two institutions.

Upload Pipeline

The upload pipeline should be explicit. First, validate the intended file name, type, size, and user. Second, create a storage key generated by the server. Third, give the client a signed upload URL. Fourth, verify the uploaded file metadata. Fifth, create a pending note and processing job. Sixth, extract text and preview data. Seventh, send it to moderation.

POST /api/uploads/presign
  -> validate user, mime type, size
  -> create storage key and UploadedFile ledger row
  -> return signed PUT URL

client uploads file directly to object store

POST /api/uploads
  -> verify storage key belongs to user
  -> verify file size and type
  -> create NoteVersion and Note in PENDING state
  -> enqueue processing and moderation jobs

Moderation System

Build moderation before growth. The current code has the right instinct: staff actions are transaction-wrapped and recorded as moderation events. Expand that into a full review console with reasons, duplicate warnings, text extraction preview, report backlog, and audit log search.

Moderation featureWhy it belongs earlyAI assist?
Approve, reject, hide, restoreBasic lifecycle control.Suggest likely decision, never auto-publish at first.
Report resolutionRequired once comments and public notes exist.Summarize report context and prior actions.
Duplicate detectionPrevents ten copies of the same PYQ file.Use file hash plus semantic title and chunk similarity.
PII and unsafe content detectionProtects students and institutions.Flag emails, phone numbers, IDs, answer-key leaks, spam.
Copyright risk taggingCampus resources can include copyrighted books or paid material.Flag suspicious source patterns for human review.

Storage Safety

Do not let the client invent arbitrary storage keys. Keep server-generated keys, sanitize file names, track uploaded file rows, and verify the final object. Public URLs are fine only for approved material; otherwise use short-lived signed download URLs.

Trust rule
The safest upload flow is not the one with the fewest steps. It is the one where every step can be explained, audited, retried, and revoked.
06 · Search, Community, And AI

Search, Community, And AI

AI should arrive after the corpus exists. Search proves the material is structured; AI then becomes a better way to navigate that structure.

Search In Three Layers

LayerUse caseImplementation
FiltersSemester, subject, course, tag, file type, owner, saved.Prisma queries and indexed relational fields.
Full-text searchExact phrases and multi-word course intent.Postgres tsvector, websearch_to_tsquery, ranked ids, fallback to substring fields.
Semantic searchAsk for concepts even when words differ.Embeddings on DocumentChunk records with hybrid ranking.

Community Features Worth Building

Community should reinforce study value, not become generic social noise. Keep comments attached to notes, collections attached to exam sprints or courses, rooms attached to focused sessions, and leaderboards attached to useful contribution, not raw activity.

  • Comments and one-level replies: enough for Q&A without turning every note into a forum thread.
  • Collections: bundles for exam sprints, unit packs, lab references, and public share links.
  • Study rooms: lightweight focus sessions tied to courses or exam windows.
  • Leaderboards: count approved contributions, helpful comments, ratings from others, and downloads excluding self-downloads.
  • Notifications: moderation decisions, comment replies, collection updates, course follower updates, and digest emails.

AI Feature Roadmap

PriorityFeatureHow it should work
P0Upload metadata assistantSuggest title, description, tags, course, unit, and PYQ flag from file name plus extracted first pages.
P0AI roadmapUse course, syllabus, weak topics, saved notes, and approved resources to generate day-by-day plans.
P0Exam sprintRank high-yield topics, produce a crash plan, and attach matching PYQs or notes.
P1Ask your notesAnswer questions from approved DocumentChunks only; cite note title, page, and chunk id.
P1Flashcard and quiz generatorGenerate cards from a note section; track confidence and resurface weak items.
P1Semantic duplicate detectorCompare upload text and embeddings against existing course notes before moderation.
P2Moderator copilotSummarize file content, risk flags, duplicate matches, and recommended moderation reason.
P2Institution insightsShow weak courses, upload gaps, high-demand topics, unanswered questions, and moderation backlog.
P3Personal study coachUse saved notes, tasks, room sessions, quizzes, and deadlines to adapt weekly plans.

AI Architecture

Keep prompts out of route handlers. Create feature-specific AI modules with typed inputs, structured outputs, model/provider settings, retries, and test fixtures. Every model response should be parsed through Zod. Every production call should create an AIUsageEvent and include a feature name, user id, institution id when present, provider, model, status, latency, and estimated token or request cost.

lib/server/ai/
  providers.ts          provider selection and retries
  structured.ts         schema-bound JSON generation
  retrieval.ts          hybrid search and citation assembly
  features/
    roadmap.ts
    exam-plan.ts
    note-suggestions.ts
    ask-notes.ts
    moderation-assist.ts
  evals/
    fixtures.ts         stable inputs and expected properties

AI Guardrails

  • Require citations for note-grounded answers; no citation, no confident answer.
  • Separate creative study planning from factual Q&A. Planning can be flexible; Q&A must be source-bound.
  • Rate-limit by user, IP, and institution. Track quota before calling the model.
  • Use background jobs for document extraction, embeddings, duplicate detection, and batch summaries.
  • Keep an eval set for every AI feature. Test schema validity, citation presence, refusal behavior, and latency budget.
07 · Frontend And Delivery Plan

Frontend And Delivery Plan

The frontend should feel like a study operating system: dense, calm, fast, and obvious. Avoid making the signed-in app look like a landing page.

Design Direction

Use a restrained product UI with a strong app shell, predictable navigation, and focused work surfaces. Keep cards for repeated items such as notes, tasks, collections, and queue rows. Use tables for moderation and admin analytics. Use drawers or side panels for note detail, citations, upload progress, and AI conversations.

SurfaceDesign ideaImportant state
DashboardStudy cockpit: tasks, current sprint, saved notes, trending course resources, room schedule.Empty account, verified account, exam week, offline or failed fetch.
LibraryFaceted browser with search, filters, quick preview, save, rate, download, collection add.No results, semantic results, pending note hidden, citation jump.
UploadWizard: file, metadata, AI suggestions, preview, policy confirmation, moderation status.Invalid mime, too large, duplicate warning, upload retry.
AI RoadmapsInputs on left, generated plan on right, linked resources and progress tracking.Quota hit, no resources found, regenerate with constraints.
Ask NotesChat panel grounded in selected course or collection with visible citations.No source answer, conflicting notes, citation unavailable.
Review QueueDense staff table plus detail pane: file preview, extraction, risk flags, actions.Already reviewed, duplicate, report escalated.
Institution AdminAnalytics dashboard: uploads, approvals, active courses, AI usage, storage, reports.Quota warning, plan inactive, domain unverified.

Component Rules

  • Use Server Components for data-heavy route shells, and client components for forms, search controls, optimistic saves, drawers, and command palettes.
  • Keep note cards stable in size. Do not let ratings, tags, or long course names resize the entire grid.
  • Use icons for repeated actions: save, upload, search, filter, report, approve, reject, hide, restore.
  • Make keyboard search a first-class workflow. Students should jump from dashboard to any note, course, room, or collection without browsing the sidebar.
  • Show source confidence in AI answers. Citations should be visible, clickable, and tied to note previews.
  • Design mobile as a real product, not a collapsed desktop table. Library search, upload status, and AI roadmaps must work on phone screens.

First 30 Days

Week 1: product contract, repo, schema, auth, institution and course seed data. Week 2: upload, storage, moderation, library basics. Week 3: search, saves, ratings, downloads, dashboard, notifications. Week 4: AI metadata, roadmap, exam plan, admin review polish, tests, pilot checklist.

Do not wait until the end to add tests. The current repo already benefits from unit tests for pure logic and Playwright specs for core flows. Preserve that habit from the first feature.

08 · Quality Bar And Launch

Quality Bar And Launch

A smarter rebuild is not finished when the features exist. It is finished when unsafe paths fail closed, AI behavior is measured, and a pilot college can trust the workflow.

Test Matrix

Security Checklist

  • HTTP-only session cookie, token hashing, expiry, and sign-out cleanup.
  • Server-side role checks for every moderator, admin, and institution route.
  • Rate limits for auth, uploads, comments, downloads, reports, and every AI endpoint.
  • Storage keys generated only by the server; path traversal rejected.
  • Mime type, size, and ledger verification before creating public note records.
  • Report and moderation events retained for audit history.
  • AI prompts avoid secrets and raw unnecessary personal data.
  • Public collection and note routes enforce visibility rules.

Launch Sequence

Launch with one pilot college or one department, not a vague global campus market. Seed 30-50 high-quality notes across 5-8 courses, recruit 5-10 contributors, and appoint at least one moderator. Make the first pilot about weekly retention and upload quality, not vanity signup counts.

Institution valueAdmin can see useful upload, usage, and risk metrics.Would a college pay for governance?

The Final Shape

The better ClassVault is not simply faster or more AI-heavy. It is more legible. Students understand where answers came from. Moderators understand what changed. Institutions understand usage and risk. Developers understand where business logic lives. That clarity is what makes the app easier to scale.

Monday morning start
Write the product contract, draw the schema, seed the first courses, and build the upload-to-moderation path before anything else. Once trusted notes exist, search, community, and AI all become easier to build well.

Glossary

Trusted corpus - The approved, permissioned set of notes and chunks that search and AI are allowed to use.

DocumentChunk - A piece of extracted note text with page and section metadata, used for semantic search and citations.

Hybrid search - Ranking that combines exact full-text matches, relational filters, and vector similarity.

AIUsageEvent - A durable record of each model call for quota, cost, debugging, and abuse analysis.
